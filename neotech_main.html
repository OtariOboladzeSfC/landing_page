What is this document about?
This is a landing page for everyone interested in general and technical aspects of NEOTEC project (explained as easy as possible). NEOTEC project has many files: from online shared docs and password manager app to offline documentations and deployment software and even training videos, so this single page will connect them all together.

To put it simple: a random person must be able to read this single document and understand where the project comes from, what is its current state and which features to expect in the future. Primarily intended for NEOTEC team members to understand general ideas behind design structure without diving deep into technical details.
Why this format?
As I believe that docs must be accessible offline and it must be more readable than .txt or .md for broader public, the choice stopped at HTML page, and even the simpleset HTML webpage gives the possibility to express any kind of medium to get the most out of tech doc (plain and formatted text, pics, hyperlinks, etc.) There will be eventually many more types of files related to the project, but they will be simply referred along the way.

Project goal
Odourcollect is a project that is already accomplished: it has webapp, android/ios apps, admin panel, database and it already met its deadline back in 2020.
Odourcollect project finished successfully.

NEOTEC is a project that aims to take Odourcollect and extend its functionalities by using its database and add more features, give it more flexibility for integration with industry and make it more extensible to other projects.
Why redesign?
Even though Odourcollect is a successful project, to make it reusable there is need to redesign it.

Odourcollect is designed by many developers throughout the years by different technologies, different methodologies and each of them without documenting any thought process behind them.

NEOTEC will be made to be extensible, easy to respond to any future changes and flexible enough to be part of bigger family of EU projects.
Difference between NEOTEC and Odourcollect
For the end user it does not really matter which technology is used behind the scenes to make an app work, but here is the catch: Odourcollect is a monolithic structure - it is designed to be Odourcollect and nothing more and it fights back for any structural changes because it does not want to be anything else than Odourcollect.

But what if you want to change database?
A backend framework or even programming language?
What if IoT devices want to read/write data on the server?
What if that one guy who maintains Vue framework suddenly dies while eating too much sushi and you have to change frontend because no one maintains it anymore?
(Okay let that guy live, he is a good programmer)

And so here is the NEOTEC: it will decouple everything from everything, make changes painless and create flexible app that can be deployed anywhere.

Development
App is developed on Docker and AWS ECR is used to save app versions.
Why? - NEOTEC is containerized and docker assures that app will be consistent on any platform, and AWS ECR is reliable platform and there is no need for private hub.
(app will follow A.B.C. versioning style where A is major release, B is added new feature and C is bug fix).

See password file for Docker Hub access credentials.
Production
App will be deployed on AWS ECS.
Why? - we do not have any infrastructure at SfC and even if we had, I can not guarantee safety of the app at that level as they do it at AWS. AWS/Microsoft Azure/Google Cloud are our best bet by now.

Database will be accessed by ORM.
Why? - this gives us possibility to entirely swap MySQL database to SQLite or PostgreSQL or whatever and still maintain code as if nothing happened, so app will not be database language dependent. Also, ORM gives the power to object-oriented developer to talk to database without knowing SQL queries.

Backend will be written with Laravel.
Why? - PHP developers are easiest to find if someone needs to review the code, and Laravel solves ORM, API, Routes, and server side rendered HTML together with many more problems.

Communication to server will be RESTful.
Why? - REST gives the possibility to use the server for web app, mobile app, IoT communication and anything that can send HTTP request to it.

Front end will be chosen between Vue and React.
Why? - Vue has shortest learning curve and React is easy to extend as React Native for mobile devices.

Main requirements
Here are the core functionalities of what a user should accomplish within the app:

Basic
1. Report odors
2. Beeing able to view, modify and remove own reported odors
3. Filter odor reports marked by other users by time and location

'Report odors' means selecting and combining:
  • 'odor type and subtype'
  • 'odor intensity'
  • 'odor niceness'
  • 'odor duration'
  • 'odor source' (optional)
  • 'comments' (optional)
Analysis
Module 'Analysis' consists of two parts: 'Basic Analysis' and 'Advanced Analysis', its structure and way of implementation is still under consideration.

API and integration
module 'Industrial Integration' and 'Data API' is incorporated within the app architecture as whole system is designed and meant for RESTful communication.
Necessary vs nice to have features
From the part of users:

NEEDED:
Report, view, modify and remove odors.
Manually filter odor cases by time and space.
Visually verify odor cases on the map.
See legal info about reported cases.

NICE TO HAVE:
Map layers for weather, wind direction and atmospheric pressure.
Notifying users about recently marked odors around them.
Download user private data.


On the part of admin:

NEEDED:
Review and confirm new cases.
Being able to modify odor types.
Add new odor intensity/odor niceness/odor duration on backend.
Download odor cases as CVS or XLSX file.

NICE TO HAVE:
(this part needs further documentation)
Multiple levels of access
There will be at least three types of users:

Regular users - users who provide information about odors and view other reported odors.
Authorities - who will be only able to view reported odors but will not be able to provide any new case.
IoT devices - devices that can report and view odors but only as machine code, these will not be human-readable.
